## Smaug Architecture (Complete Understanding)

### The Flow
1. **Fetch Phase** (processor.js → bird CLI)
   - Uses `bird` CLI to fetch Twitter bookmarks/likes
   - Expands t.co shortened links
   - Extracts content from linked pages (articles, GitHub repos, tweets, X articles)
   - For GitHub: fetches repo metadata, stars, README
   - For articles: extracts title, author, content
   - For X articles: uses bird CLI (requires JS rendering)
   - Saves to `pending-bookmarks.json`

2. **Process Phase** (job.js → Claude Code)
   - Reads pending bookmarks
   - For small batches (<8): sequential processing via Claude Code
   - For large batches (≥8): parallel subagent processing (uses Haiku for cost savings)
   - Claude analyzes each bookmark and determines:
     - Title/summary (not generic)
     - Category (github, article, podcast, youtube, video, tweet)
     - Action: file (separate .md in knowledge/), capture (bookmarks.md only), transcribe (flag for later)
   - Uses `.claude/commands/process-bookmarks.md` for instructions
   - Writes to bookmarks.md (EDIT tool, never Write)
   - Files GitHub repos to `knowledge/tools/`, articles to `knowledge/articles/`, etc.

### Key Technologies
- **bird CLI** by Peter Steinberger: Twitter API wrapper that needs auth_token + ct0 cookies
- **Claude Code**: AI-powered analysis and categorization
- **dayjs**: Date formatting
- **Node.js 20+**: Runtime

### Configuration
- `smaug.config.json`: auth tokens, archive paths, timezone, model selection, categories
- `.claude/commands/process-bookmarks.md`: processing instructions for Claude
- `.state/pending-bookmarks.json`: bookmarks waiting to be processed
- `.state/bookmarks-state.json`: processing state/metadata
